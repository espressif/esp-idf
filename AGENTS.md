# AGENTS.md

This file provides guidance to WARP (warp.dev) when working with code in this repository.

## Repository Overview

ESP-IDF (Espressif IoT Development Framework) is the official development framework for Espressif SoCs (ESP32, ESP32-S2, ESP32-S3, ESP32-C2, ESP32-C3, ESP32-C6, ESP32-H2, ESP32-P4, etc.). It uses a CMake-based build system orchestrated via the `idf.py` CLI tool.

## Environment Setup

Before running any build commands, the ESP-IDF environment must be activated in the current shell:

```
# Windows
.\install.ps1       # first time only
.\export.ps1        # every shell session

# Linux/macOS
./install.sh        # first time only
source export.sh    # every shell session
```

## Build Commands

All build operations go through `idf.py`, which wraps CMake + Ninja:

```
idf.py set-target <chip>         # Set target SoC (esp32, esp32s2, esp32c3, etc.)
idf.py menuconfig                # Open Kconfig text-based configuration menu
idf.py build                     # Build app, bootloader, and partition table
idf.py -p PORT flash             # Flash to device
idf.py -p PORT monitor           # Serial monitor (Ctrl-] to exit)
idf.py flash monitor             # Build + flash + monitor in one pass
idf.py app                       # Build only the app (skip bootloader)
idf.py app-flash                 # Flash only the app
idf.py -B build_dir build        # Use custom build directory
```

Supported targets: `esp32`, `esp32s2`, `esp32c3`, `esp32s3`, `esp32c2`, `esp32c6`, `esp32h2`, `esp32p4`. Preview targets: `linux`, `esp32c5`, `esp32c61`.

## Project Structure

A typical ESP-IDF project has this structure:
```
project/
├── CMakeLists.txt          # Must include($ENV{IDF_PATH}/tools/cmake/project.cmake)
├── main/
│   └── CMakeLists.txt      # idf_component_register(SRCS ... INCLUDE_DIRS ...)
├── components/             # Optional project-local components
├── sdkconfig               # Generated by menuconfig
└── sdkconfig.defaults      # Default config overrides
```

The root `CMakeLists.txt` for any project must contain:
```cmake
cmake_minimum_required(VERSION 3.16)
include($ENV{IDF_PATH}/tools/cmake/project.cmake)
project(my_project)
```

## Component System (Critical Architecture)

ESP-IDF is built around a **component-based architecture**. Every library, driver, and piece of framework code is a "component" under `components/`.

### Component Registration

Each component has a `CMakeLists.txt` that calls `idf_component_register()`:
```cmake
idf_component_register(
    SRCS "source.c"
    INCLUDE_DIRS "include"
    REQUIRES other_component           # Public dependency
    PRIV_REQUIRES private_dep          # Private dependency
    REQUIRED_IDF_TARGETS esp32 esp32s3 # Restrict to certain targets
    LDFRAGMENTS "linker.lf"
    WHOLE_ARCHIVE                      # Link as --whole-archive
)
```

- `REQUIRES` = public dependency (headers exposed to dependents)
- `PRIV_REQUIRES` = private dependency (only used internally)
- Components without `SRCS` become interface-only (header-only) libraries

### Component Dependency Layers (G0 → G1 → higher)

**G0 (lowest level):** `soc`, `hal`, `esp_rom`, `esp_common`, `riscv`/`xtensa`
- `soc`: Register structures, addresses, pin definitions, `SOC_*` capability macros
- `hal`: Hardware abstraction. Two sub-layers:
  - **LL (Low-Level)** `hal/<periph>_ll.h`: Static inline functions mapping to registers. Target-specific, located at `components/hal/<target>/include/hal/<periph>_ll.h`
  - **HAL** `hal/<periph>_hal.h`: Higher-level operations combining LL calls. OS-independent.
- `esp_rom`: ROM function headers, linker scripts, patches

**G1:** `esp_hw_support`, `esp_system`, `freertos`, `newlib`, `spi_flash`, `log`, `heap`, `esp_mm`
- `esp_system`: Startup, panic handling, watchdogs, reset reason
- `esp_hw_support`: Interrupt allocation, sleep, clocks, MAC settings (facilitates HW access, doesn't own resources)

**Higher-level:** Peripheral drivers (`esp_driver_*`), networking (`esp_wifi`, `esp_eth`, `lwip`), protocols (`mqtt`, `esp_http_client`), storage (`nvs_flash`, `fatfs`, `spiffs`), etc.

### Shared Register Protection (HAL Layer)

When LL functions access shared registers (e.g., reset/clock control), they must be called within critical section macros:
- `PERIPH_RCC_ACQUIRE_ATOMIC(module, ref_count)` — protects + increments ref counter
- `PERIPH_RCC_RELEASE_ATOMIC(module, ref_count)` — protects + decrements ref counter
- `PERIPH_RCC_ATOMIC()` — protects only (no ref counting)

These macros are from `esp_private/periph_ctrl.h`.

### Target-Conditional Code

Components frequently use Kconfig-driven `CONFIG_*` and `SOC_*` macros for target-specific compilation:
```cmake
if(CONFIG_SOC_ADC_SUPPORTED)
    list(APPEND srcs "adc.c")
endif()
```

The `IDF_TARGET` build variable and `idf_build_get_property(target IDF_TARGET)` are used in CMake to branch per-chip.

## Kconfig Configuration

ESP-IDF uses Kconfig for build-time configuration. The root `Kconfig` is at the repo root. Each component can provide:
- `Kconfig` — component-level config options (under "Component config" menu)
- `Kconfig.projbuild` — project-level config options (top-level menu)

`SOC_*` capabilities come from `components/soc/<target>/include/soc/soc_caps.h` and are auto-generated into Kconfig via `tools/gen_soc_caps_kconfig/`.

## Testing

### pytest-embedded (Primary CI test framework)

Tests use [pytest](https://github.com/pytest-dev/pytest) with [pytest-embedded](https://github.com/espressif/pytest-embedded). Test files must be named `pytest_*.py`.

```
# Run a specific test (requires hardware or QEMU)
pytest --target esp32 path/to/pytest_foo.py

# Collect tests without running
pytest --collect-only --target esp32 path/to/pytest_foo.py
```

Test markers: `@pytest.mark.esp32`, `@pytest.mark.supported_targets`, `@pytest.mark.generic`, `@pytest.mark.qemu`, etc.

### Component test apps

Components have test apps under `components/<name>/test_apps/`. These are standalone ESP-IDF projects with their own `CMakeLists.txt`, `pytest_*.py`, and `sdkconfig.ci.*` variants:
```
components/esp_driver_gpio/test_apps/
├── gpio/
│   ├── CMakeLists.txt
│   ├── pytest_gpio.py
│   ├── sdkconfig.ci.iram_safe
│   ├── sdkconfig.ci.release
│   └── sdkconfig.defaults
└── .build-test-rules.yml
```

Build and run a component test app:
```
cd components/esp_driver_gpio/test_apps/gpio
idf.py set-target esp32
idf.py -B build_esp32_default build
```

### Unit Test App (legacy, being deprecated)

Located at `tools/unit-test-app/`. Runs unit tests from component `test/` subdirectories:
```
cd tools/unit-test-app
idf.py -T <component_name> build     # Build tests for specific component
idf.py -T all build                  # Build all component unit tests
```

### Build/Test Rules

`.build-test-rules.yml` files control which apps build for which targets and under what conditions in CI. They reference `SOC_*` capabilities and `IDF_TARGET`.

## Code Style and Linting

### C/C++ formatting
Uses `astyle_py` (Artistic Style) with OTBS style, 4-space indent. Run via:
```bash
# Format specific files (Linux/macOS)
bash tools/format.sh path/to/file.c

# Or via pre-commit
pre-commit run astyle_py --files path/to/file.c
```

### Python
- **flake8** with max line length 160 (`--config=.flake8`)
- **reorder-python-imports** (py38+)
- **mypy** for type checking

### Pre-commit hooks
The repo uses extensive pre-commit hooks (`.pre-commit-config.yaml`):
```
pre-commit run --all-files          # Run all hooks
pre-commit run <hook-id> --files <files>  # Run specific hook
```

Key hooks: `trailing-whitespace`, `flake8`, `astyle_py`, `check-copyright`, `codespell`, `cmake-lint`, `check-kconfig-files`, `shellcheck`, `conventional-precommit-linter` (commit message format).

### Copyright headers (SPDX)

All source files require SPDX copyright headers. Config is in `tools/ci/check_copyright_config.yaml`.

For C/C++:
```c
/*
 * SPDX-FileCopyrightText: 2024 Espressif Systems (Shanghai) CO LTD
 *
 * SPDX-License-Identifier: Apache-2.0
 */
```

For Python:
```python
# SPDX-FileCopyrightText: 2024 Espressif Systems (Shanghai) CO LTD
# SPDX-License-Identifier: Apache-2.0
```

Default license is `Apache-2.0`. Examples and test files use `Unlicense OR CC0-1.0`.

### EditorConfig

The `.editorconfig` specifies: 4-space indent (C, Python, CMake), 2-space indent (YAML, shell), LF line endings, UTF-8, max line lengths (Python: 119, CMake: 120).

## Warp Workflows

Pre-built workflows for common tasks are in `.warp/workflows/`. Access them in Warp via `CTRL+SHIFT+R` (Workflow Search) and type the workflow name or tags.

### OpenThread Border Router & Radio Co-Processor

The OTBR + RCP setup requires two devices: a Wi-Fi SoC (e.g. ESP32-C6) for the border router and an 802.15.4 SoC (e.g. ESP32-H2) for the radio, connected via UART.

Available workflows:
- **OT Border Router - Build** (`otbr_build.yaml`) — Build OTBR firmware. Param: `target` (default: `esp32c6`)
- **OT Border Router - Flash** (`otbr_flash.yaml`) — Flash OTBR + open monitor. Param: `port`
- **OT Border Router - Configure Wi-Fi** (`otbr_configure.yaml`) — Open menuconfig for Wi-Fi/radio settings
- **OT Radio Co-Processor - Build** (`rcp_build.yaml`) — Build RCP firmware. Param: `target` (default: `esp32h2`)
- **OT Radio Co-Processor - Flash** (`rcp_flash.yaml`) — Flash RCP firmware. Param: `port`
- **OT Border Router + RCP - Full Setup** (`otbr_rcp_full_setup.yaml`) — Build and flash both RCP then OTBR in one pass

## Conventions When Modifying Code

- New peripheral drivers go in `components/esp_driver_<periph>/`, not the legacy `components/driver/` (which contains deprecated wrappers)
- Target-specific implementations use subdirectories named by target: `components/<comp>/<target>/` or `components/<comp>/port/<target>/`
- Use `idf_component_optional_requires()` for conditional dependencies based on Kconfig
- The `linux` target allows building and running certain components on the host for testing without hardware
- `sdkconfig.ci.*` files are CI-specific config overlays; `sdkconfig.defaults` is the normal project default
- Commit messages follow the [Conventional Commits](https://www.conventionalcommits.org/) format (enforced by pre-commit)
